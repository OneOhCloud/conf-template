# conf-template conventions

This repo ships `sing-box` config templates to OneBox (and any other client that
pulls them off the CDN). Before you edit anything, read this file.

## The contract

1. **`conf/1.13.8/zh-cn/*.jsonc` is the canonical source.** Hand-edit only
   these four files.
2. **`conf/1.13/zh-cn/*.jsonc` and `conf/1.12/zh-cn/*.jsonc` are generated.**
   Running `pnpm generate` overwrites them from the canonical files. Do not
   edit them directly — your changes will be blown away on the next generate
   run.
3. **Every generate run is validated.** Static checks (reference integrity,
   required fields, forbidden legacy fields) + optional `sing-box check`
   (under `--strict` / `SING_BOX_BIN`) both run before any file is written.
   A validation failure aborts the run and touches nothing.

## Why this shape

We tried hand-editing each version independently. Drift set in immediately —
`dns.rules` got lost on one version, `sniff_override_destination` stuck around
on another, rule_set references diverged. Every release someone discovered a
new inconsistency.

The generator collapses "edit N files" to "edit one file"; the validator
collapses "hope we remembered everything" to "every run, every file, checked".

## Usage

```bash
# Install deps (one-time)
pnpm install

# Normal edit cycle: modify conf/1.13.8/zh-cn/<variant>.jsonc, then:
pnpm generate

# Full check with sing-box check on every file (requires a sing-box binary)
SING_BOX_BIN=/path/to/sing-box pnpm generate:strict

# Dry run — validate only, don't write
pnpm check
```

After `pnpm generate`, `git diff` shows you exactly what changed across all
12 output files (canonical × 3 + derived × 2 × 4 = 12).

## Version transformations

Currently all three versions (`1.12`, `1.13`, `1.13.8`) are structurally
identical — the derived versions are an exact copy of canonical. This holds
because every feature in the canonical template has been supported since
`sing-box` 1.11 (route-rule `sniff` action, `dns.rules` split routing, fakeip
transport type, `default_domain_resolver`).

When a future canonical edit introduces something that's **not** backward
compatible with 1.12 or 1.13, add the version-specific subtraction in
`scripts/generate.ts::transform()` and document the reason inline. Never edit
the derived files by hand to "fix" a version-specific issue — that bypasses
the generator contract and the drift will come back.

## Validator (what's checked)

1. **Static structural** — on every run:
    - Top-level blocks (`dns`, `inbounds`, `route`, `outbounds`) exist.
    - Every `dns.rules[].server` and `dns.final` references a tag in
      `dns.servers`.
    - Every `dns.rules[].rule_set` and `route.rules[].rule_set` references a
      tag in `route.rule_set`.
    - Every `route.rules[].outbound`, `route.final`, and
      `outbound[].outbounds[]` references an existing outbound tag (including
      implicit `direct` / `block`).
    - **1.13.8**: legacy `sniff` / `sniff_override_destination` inbound fields
      are forbidden. TUN variants must have
      `{"inbound":"tun","action":"sniff"}` as `route.rules[0]`.
    - TUN variants must have a `type: "tun"` inbound. Mixed variants must
      not.
    - **-rules variants**: `dns.rules` must route CN traffic (`geoip-cn` /
      `geosite-cn`) to a `system` transport. Forgetting this was how the
      `www.qq.com → overseas IP` regression got in.

2. **`sing-box check`** (optional, `--strict` or `SING_BOX_BIN=...`) — runs
   the real sing-box parser against a patched copy of each config. The
   generator injects a stub outbound into empty `auto.outbounds` /
   `ExitGateway.outbounds` placeholders (OneBox fills these at runtime from
   user subscriptions) so the check doesn't fail on "missing tags". The real
   on-disk file is never modified during check.

## Output format

Every generated `.jsonc` starts with a 5-line header:

```jsonc
// AUTO-GENERATED by scripts/generate.ts — do not edit by hand.
// Source: conf/1.13.8/zh-cn/<variant>.jsonc (canonical)
// Regenerate: pnpm generate
// Variant: <variant>   Version: <version>
// CDN path: https://cdn.jsdelivr.net/gh/OneOhCloud/conf-template@dev/conf/<version>/zh-cn/<variant>.jsonc
```

Followed by a stable-formatted JSON body (4-space indent). Inline comments
in the canonical file are **not preserved** in derived outputs — they survive
only in the canonical file itself, and even there they're stripped the next
time the generator round-trips through `JSON.parse`/`JSON.stringify`. Treat
them as editor hints, not load-bearing documentation. For anything that
needs to survive, put it in this `CONVENTIONS.md`.

## Adding a new variant / version

1. **New variant** (e.g. `tun-bypass`): add the canonical file under
   `conf/1.13.8/zh-cn/`, add the tag to `VARIANTS` in `scripts/generate.ts`,
   add variant-specific validator rules if needed. `pnpm generate` emits all
   versions.

2. **New version** (e.g. `1.14`): add to `DERIVED_VERSIONS` in
   `scripts/generate.ts` and add a new `case` in `transform()` describing how
   to derive it. If 1.14 becomes the new canonical (e.g. after a major
   sing-box upgrade), move files to `conf/1.14/zh-cn/`, update `CANONICAL`,
   and make `1.13.8` a derived version (possibly with legacy compatibility
   overlays).

3. **New DNS rule or domain**: edit only `conf/1.13.8/zh-cn/<variant>.jsonc`.
   Run `pnpm generate`. Review `git diff`. Commit.

## Files

```
conf-template/
├── package.json              — pnpm config + generate scripts
├── tsconfig.json
├── CONVENTIONS.md            — this file
├── scripts/
│   └── generate.ts           — entry point (single file, ~450 lines)
├── conf/
│   ├── 1.13.8/zh-cn/*.jsonc  — CANONICAL (hand-edit)
│   ├── 1.13/zh-cn/*.jsonc    — DERIVED (generator output)
│   └── 1.12/zh-cn/*.jsonc    — DERIVED (generator output)
└── database/                 — separate concern, not touched by generate
```

## The sing-box check limitation

The generator runs `sing-box check` against every emitted file, which is
stronger than any static check we could write. But it has one important
caveat: **we run whatever `sing-box` binary is in `$SING_BOX_BIN` or
`$PATH`, not a version-specific one**. So when the canonical is validated
with sing-box 1.13.8 and that binary accepts it, we only know the output is
1.13.8-compatible — we have not proven it's 1.12-compatible.

If a sing-box feature lands between 1.12 and 1.13.8 that is not backward
compatible, the 1.12 output may pass check (because we're running the 1.13.8
binary against it) but fail at runtime on a 1.12 client. For now we accept
this risk because:

- The features currently in the canonical all predate 1.12.
- OneBox's kernel version is pinned (one binary version ever runs the config).
- Old templates (`conf/1.12/`) are only served for backward compat with older
  OneBox builds, which also ship their own sing-box binary.

If the risk materializes, add a second `SING_BOX_BIN_1_12=/path/to/sing-box-v1.12`
env var and run the 1.12 derived outputs against that binary too. The
generator's check loop already keys on `version` in its output labels.
